Storage account details saved for tenant {{.TenantID}}

Complete the two steps below to finish deployment.

Step 1 — Assign the Storage Blob Data Contributor role

Run this command in Azure CLI, scoped to the specific container for least-privilege access:

az role assignment create \ --assignee-object-id $(az ad sp show --id {{.ClientID}} --query id -o tsv) \ --role "Storage Blob Data Contributor" \ --scope {{.ResourceID}}/blobServices/default/containers/{{.ContainerName}} \ --assignee-principal-type ServicePrincipal

Run this command while logged into the target tenant ({{.TenantID}}) with an account that has Owner or User Access Administrator on the storage account. Scoping RBAC to the container (rather than the storage account) limits the blast radius to this container only. The timestamp writer will automatically begin writing once the role assignment propagates (may take a few minutes).

Step 2 — Verify the AKS workload identity annotations

Confirm deploy/serviceaccount.yaml is annotated with your UAMI client ID (not the app client ID below). The Identity Binding RBAC check uses the UAMI client ID.

# deploy/serviceaccount.yaml — annotation must use UAMI client ID azure.workload.identity/client-id: "<YOUR_UAMI_CLIENT_ID>" # from Step 2 of azure-setup.md azure.workload.identity/tenant-id: "<SOURCE_TENANT_ID>"

Confirm deploy/configmap.yaml has the Entra app client ID:

# deploy/configmap.yaml — AZURE_CLIENT_ID must be the Entra app client ID AZURE_CLIENT_ID: "{{.ClientID}}" AZURE_TENANT_ID: "{{.TenantID}}"

The timestamp writer reads storage account configuration directly from the shared database — no additional changes to deploy/configmap.yaml are required.
